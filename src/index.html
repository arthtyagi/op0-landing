// the object

Every agent change is one signed Proposal.

Not a thread. Not a checklist. One signed Proposal — intent, diff, preview, policy, rollback target, approval — that reviewers sign, operators revert, and auditors read for as long as the surface exists.

one signed object

Reviewers sign the same object operators can revert.

  • intent

    why this change exists

  • diff

    what changed

  • preview

    where to see it run

  • policy

    the decision that approved it

  • rollback

    what it reverts to, by name

  • approval

    who signed it, when

// the loop

Signal in. Shipped product out.

Four steps. Every one writes evidence to the Runtime Workspace. Every one has a name agents can read.

  1. 01

    A signal arrives.

    Replay, error, annotation, support note, or analytics anomaly — a typed signal queued for triage.

    # signal inbox · op0.dev/i

    replay checkout-abandon · safari 17 · 3 events

    error TypeError on unload · src/replay.ts:142

    annotation "button overlaps on <375px" · @arth

  2. 02

    op0 writes a Context Bundle.

    op0 writes a Context Bundle in the Runtime Workspace so agents read replays, events, and state in one place.

    context-bundle · op0-7f3a

    events: 3 · replay: safari-17 · git: src/replay@main

    state: session_8821 · policy: default-rig@2

    prior: op0-7f29 (last good)

  3. 03

    An agent drafts a Proposal.

    Bounded by the active Rig. An agent drafts a Proposal — diff, preview, policy, and rollback target on one object.

    proposal · op0-7f3a · draft

    +128 −64 · src/replay.ts

    preview → preview.op0.dev/p/7f3a

    rollback → op0-7f29 (named)

  4. 04

    You ship a manifest, a PR, or both.

    Delivery emits a signed Surface Manifest, a Git PR, or both. Every deploy names what it would revert to.

    git delivery

    PR #412 · 2 checks ✓

    reviewers: @arth

    surface manifest

    signed · op0.dev/m/7f3a

    revert → op0-7f29

// where op0 sits

We don't replace your agent. We ship its commit.

Code-writing agents write the code. Always-on repo agents keep the repo healthy. Analytics-driven agents read user behavior and propose changes from it. op0 owns the part between commit and shipped product.

  • above

    Whatever writes the code.

    IDE agents, CLI agents, autonomous engineers, always-on repo agents, agents that read user analytics and propose code from it, agent teams. op0 picks up the commit, the PR, or the surface request.

  • op0

    Proposals, Manifests, Rollback Targets.

    One signed object per change. Diff, preview, policy decision, rollback target. Delivered to the surface, recorded in audit, reversible by name.

  • below

    Whatever runs underneath.

    Sandboxes, browsers, source adapters, edge delivery, MCP transports. op0 abstracts the substrate behind one command surface — same API in HTTP, CLI, desktop, MCP.

// the rig

The spec your agents execute against.

Edit the rig, not the agent. One file declares signals, agent definitions, capability grants, model slots, and the artifact bundle every Proposal must include.

# rig.toml · op0.dev/r

[signals]

replay, error, annotation, support, analytics

[agents]

fixer = { model = "anthropic/claude", grants = ["replay.read"] }

[output]

artifact = "proposal.v1" # diff + preview + policy + rollback

  • model slots

    swap the LLM without rewriting the agent. signed binding per run.

  • capability grants

    scoped per Source, per agent. unauthorized calls fail typed.

  • output contracts

    Proposal artifact bundle is declared, not improvised.

// capability console

Operators drive it. Agents call it.

Capabilities are not hard-coded into agents — they're installed from a registry, bound to a Rig, and revocable in one place.

discover

install
grants are scoped per agent, per Source, per Rig version.
invoke
every call creates a durable session, message, and event chain.
audit
agents and operators read the same transcript.

// architecture

How op0 stays reversible.

Three commitments, hard-coded into the runtime. They're the reason every change is auditable and every release names what it would revert to.

  1. 01

    Single command boundary.

    Every governed action goes through one typed API surface.

  2. 02

    Durable product state.

    Proposals, Versions, audit, and rollback targets survive reloads.

  3. 03

    Reversible by design.

    Every release names its rollback target before it ships.
